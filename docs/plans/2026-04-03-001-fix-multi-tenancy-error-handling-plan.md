# Harden Multi-Tenancy & Error Handling

## Overview

This plan addresses **8 HIGH-severity security and reliability issues** discovered in the codebase sweep:

1. **Missing `tenant_id` filters** in `trainingPlan.service.ts` — allows cross-tenant data access
2. **Unauthenticated WebSocket path** in `CoachAgent.ts` — accepts connections without session validation
3. **Zero error handling** across all service files — D1 query failures throw unhandled
4. **Non-null assertion on potentially-undefined** tenant settings (future crash path)
5. **Missing CORS headers** on tRPC endpoint
6. **TOCTOU race conditions** in upsert functions (read-then-write without transactions)
7. **Agent route bypassing auth middleware** in worker
8. **Hardcoded dev credentials** in `worker.tsx` auth bypass

**Priority:** P0 — Multi-tenant data leakage is a security vulnerability. Unhandled D1 errors cause 500s in production.

---

## Proposed Solution

### Part A: Multi-Tenancy Hardening (Security-Critical)

#### 1. Add `tenant_id` filters to training plan queries

**Files affected:** `src/services/trainingPlan.service.ts`

| Function | Current State | Fix |
|----------|---------------|-----|
| `getTrainingSessionsByPlan()` | No tenant filter | Add optional `tenant_id` param, filter by plan's tenant |
| `getTrainingSessionsByWeek()` | No tenant filter | Add optional `tenant_id` param, filter by plan's tenant |
| `getSessionExercisesBySession()` | No tenant filter | Add optional `tenant_id` param, filter by session's tenant |
| `getSessionExercisesGrouped()` | No tenant filter | Delegate to fixed `getSessionExercisesBySession` |
| `cloneTrainingPlanToTenant()` | Reads source plan without tenant scope | Keep as-is (system templates have NULL tenant_id) |

**Implementation approach:**
- Functions that traverse relationships (session → plan) should accept optional `tenant_id` and verify the parent entity belongs to that tenant
- For `getTrainingSessionsByPlan`, first verify the plan belongs to tenant, then filter sessions

#### 2. Add session validation to CoachAgent WebSocket

**Files affected:** `src/agent/CoachAgent.ts`

Current `onConnect` accepts `userId` and `tenantId` from query params/headers without verifying against `UserSession`. This allows impersonation.

**Fix options:**
- **Option A (Recommended):** Pass session cookie through WebSocket connection, validate against `UserSession` DO before accepting
- **Option B:** Require a short-lived WebSocket token generated by authenticated tRPC endpoint

For Phase 1, implement Option A by:
1. Reading session cookie from `ctx.request.headers`
2. Calling `UserSession.getSession()` to validate
3. Rejecting connection with 1008 if invalid

#### 3. Gate hardcoded auth bypass behind strict dev check

**Files affected:** `src/worker.tsx`

Current code:
```typescript
if (!session && import.meta.env.DEV) {
  const devSession = cookies.match(/dev_session=([^;]+)/)?.[1];
  if (devSession === "seed-user-001:seed-tenant-001") {
    ctx.session = { userId: "seed-user-001", tenantId: "seed-tenant-001" };
  }
}
```

**Fix:**
- Add additional guard: `process.env.NODE_ENV === 'development' && import.meta.env.DEV`
- Log warning when dev bypass is used
- Consider rate-limiting dev login endpoint

### Part B: Error Handling (Reliability)

#### 4. Add typed error handling to service layer

**Files affected:** All `src/services/*.service.ts`

Create a shared error utility:

```typescript
// src/services/errors.ts
export class ServiceError extends Error {
  constructor(
    public code: 'NOT_FOUND' | 'UNAUTHORIZED' | 'VALIDATION_ERROR' | 'DATABASE_ERROR',
    message: string,
    public cause?: unknown
  ) {
    super(message);
    this.name = 'ServiceError';
  }
}

export function wrapDatabaseError<T>(operation: string, fn: () => Promise<T>): Promise<T> {
  return fn().catch(error => {
    throw new ServiceError('DATABASE_ERROR', `${operation} failed`, error);
  });
}
```

Wrap all database operations:
```typescript
export async function getTrainingPlanById(db, input) {
  return wrapDatabaseError('getTrainingPlanById', async () => {
    // ... existing query
  });
}
```

#### 5. Handle TOCTOU in upsert functions

**Files affected:** `src/services/dailyWellness.service.ts`, `src/services/workoutSession.service.ts`

Current `upsertDailyWellness` has a race condition:
1. Read existing record
2. If exists, update
3. If not, insert

Between steps 1 and 3, another request could insert the same record.

**Fix:** Use D1's `ON CONFLICT` clause with Kysely's `onConflictDoUpdate`:

```typescript
export async function upsertDailyWellness(db, input) {
  const id = crypto.randomUUID();
  const now = new Date().toISOString();
  const hrv_ratio = calculateHrvRatio(input.hrv_rmssd, input.rhr);

  return db
    .insertInto('daily_wellness')
    .values({...})
    .onConflict((oc) => oc
      .columns(['tenant_id', 'user_id', 'date'])
      .doUpdateSet({
        rhr: input.rhr,
        hrv_rmssd: input.hrv_rmssd,
        // ... other fields
        updated_at: now,
      })
    )
    .returningAll()
    .executeTakeFirst();
}
```

### Part C: CORS & Headers (Infrastructure)

#### 6. Add CORS headers to tRPC handler

**Files affected:** `src/trpc/handler.ts`

Current handler returns responses without CORS headers. For browser-based clients on different origins, this blocks requests.

**Fix:** Add CORS middleware or headers:
```typescript
const corsHeaders = {
  'Access-Control-Allow-Origin': env.CORS_ORIGIN || '*',
  'Access-Control-Allow-Methods': 'GET, POST, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type, Authorization',
};

// Handle preflight
if (request.method === 'OPTIONS') {
  return new Response(null, { headers: corsHeaders });
}

// Add to all responses
return new Response(json, { headers: { ...corsHeaders, 'Content-Type': 'application/json' } });
```

## Technical Considerations

### Architecture Impacts
- **Service layer:** All services will now throw `ServiceError` instead of returning `undefined` on failure
- **tRPC routers:** Will need to catch `ServiceError` and convert to appropriate tRPC errors
- **CoachAgent:** WebSocket connections require valid session cookie

### Schema Changes
- None required (this is purely code-level hardening)

### Security Considerations
- Dev auth bypass must be impossible in production builds
- WebSocket connections must validate session before accepting
- Cross-tenant queries must be blocked at service layer

### Performance Considerations
- `ON CONFLICT` upserts are more efficient than read-then-write
- Adding tenant_id filters may require composite indexes (verify query plans)

### Rollback Plan
- Each fix is isolated to specific files
- Can deploy incrementally (Part A before Part B)
- No schema changes = no migration rollback needed
